# Product Documentation

Create factual, concise product documentation that answers the main reader questions without unnecessary storytelling.

## Workflow

1. Infer the right document scope from the request.
2. Propose a logical Markdown filename when useful.
3. Structure the content so it answers:
   - What is it for
   - Who is it for
   - How does it work
   - What business or product impact does it have
4. Adapt the structure when the topic is more technical or more operational.

## Default structure

```md
# [Feature or product area]

## Objectif

## Fonctionnement

## Cas d'usage

## Impact business / produit

## Questions fréquentes
```

## Style

- Keep paragraphs short.
- Prefer factual language over storytelling.
- Avoid unexplained jargon.
- Optimize for mixed audiences across product, business, marketing, and tech.
